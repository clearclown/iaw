プロジェクト企画書: Aether (Ether for AI Agents)

1. プロジェクト概要

プロジェクト名: Aether (コマンド名: ajj)

コアコンセプト: "Infrastructure as Workspace" (IaW)

スローガン: "Parallel Universes for AI Agents"（AIエージェントのための並行宇宙）

2. 背景と課題：AI開発のボトルネック

AIコーディングエージェント（Claude Code, GitHub Copilot Workspace等）の能力は飛躍的に向上しており、人間よりも高速にコードを実装可能です。しかし、**実行環境（Infrastructure）は依然として「シングルトン（単一資源）」**のままです。

The "Singleton Infrastructure" Problem

複数のAIエージェントを同時に走らせて並列開発を行おうとすると、以下の物理的な制約に衝突します。

ポート競合: 複数のタスクで localhost:5432 (Postgres) を奪い合い、テストが落ちる。

状態汚染 (State Pollution): Agent Aが行ったDBマイグレーションが、Agent Bのテスト環境を破壊する。

リソース枯渇: ローカルマシン上で Testcontainers を5並列で起動すると、メモリとCPUが枯渇し、開発者の作業が阻害される。

これでは、AIの「並列処理能力」を活かしきれず、結局人間が1つずつレビューしてマージする逐次処理に戻ってしまいます。

3. 解決策：Infrastructure as Workspace (IaW)

バージョン管理システム（jj）の「ワークスペース」と、コンテナ技術による「隔離環境」を 1対1で不可分なものとして結合 します。

Code Branching: ソースコードの分岐（jj workspace）

Infrastructure Branching: インフラの分岐（K8s Namespace / Remote Containers）

これらを同期させることで、AIエージェントは「ブランチを切る」という操作だけで、自分専用の、誰にも邪魔されない、クリーンな更地（Universe） を手に入れます。

4. 提供価値 (Value Proposition)

for AI Agents (自律性の最大化)

決定論的な環境: 環境構築のエラーや競合をAIに推論させる無駄を排除。「環境は常に与えられるもの」として、コーディングのみに集中させる。

コンテキスト認識: 自分のワークスペースIDに紐づいたリソース情報（DB接続文字列など）が自動注入されるため、設定ミスがなくなる。

for Developers (スケーラビリティ)

ローカルマシンの解放: コンテナ実行リソースをリモート（Kubernetesクラスタやパワフルなサーバー）にオフロードすることで、手元のMacBookは常に軽量な状態を維持。

レビュー体験の向上: PRレビュー時に「環境を立ち上げる」手間がなく、リンク一つ（あるいはコマンド一発）で動作確認が可能。
